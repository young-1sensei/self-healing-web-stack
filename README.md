Product Requirements Document (PRD)1. Project OverviewProject Name: Automated Control Plane & Self-Healing Web StackObjective: To deploy a centralized Management EC2 instance (Control Node) that automatically discovers, configures, and maintains two private Web Server instances using Terraform and Ansible.2. Target Architecture1 x Control Node: Public-facing, pre-installed with Ansible, Terraform, and Git. It acts as the "Source of Truth."2 x Web Instances: Located in private subnets, accessible only via the Control Node or the Load Balancer.1 x Application Load Balancer (ALB): Routes traffic to the Web Instances.3. Functional RequirementsFR1: Automated Infrastructure Provisioning (Terraform)Network Isolation: Create a VPC with:Public Subnets: For the Control Node and ALB.Private Subnets: For the Web Instances (No direct internet access).Security Groups:Control SG: Allow SSH (Port 22) from the user's specific IP.Web SG: Allow SSH only from the Control SG and HTTP (Port 80/443) only from the ALB SG.Dynamic Discovery: Use Terraform to tag Web Instances with Type=Managed-Webserver for Ansible dynamic inventory.FR2: Control Node Bootstrap (User Data)Upon launch, the Control Node must automatically:Install ansible, terraform, and git.Clone the project repository from GitHub.Securely pull or generate an SSH keypair used to communicate with the Web Instances.FR3: Configuration Management (Ansible)Role-Based Deployment: Apply role_common, role_nginx, and role_app to the Web Instances.Idempotency: Re-running the playbooks should ensure the "Desired State" without breaking existing services.4. Technical SpecificationsComponentSpecificationCloud ProviderAWS (Free Tier Eligible: t2.micro / t3.micro)IaCTerraform (State stored in S3/DynamoDB for "Pro" touch)Config ManagementAnsible (using aws_ec2 dynamic inventory plugin)OrchestrationAWS Auto Scaling Group (Desired: 2, Min: 2, Max: 3)OSAmazon Linux 2023 or Ubuntu 24.04 LTS5. The "Control" WorkflowThe project must demonstrate this specific lifecycle:Deployment: User runs Terraform locally to spin up the Control Node.Handoff: The Control Node's User Data script triggers a second Terraform run from within the Control Node to spin up the Web Instances.Config: The Control Node runs an Ansible playbook against the newly created Web Instances using their private IPs.6. Success Metrics (Portfolio Demo)Zero-Manual-Config: The entire stack is up and running with one initial command.Security Validation: Prove that the Web Instances cannot be pinged or SSH'd into from the public internet, but can be managed by the Control Node.Self-Healing: Terminate a Web Instance; the ASG replaces it, and a cron job on the Control Node (or an EventBridge trigger) runs Ansible to configure the new replacement automatically.7.
